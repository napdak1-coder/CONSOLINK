> 해운 콘솔사 오퍼레이터를 위한 업무 도우미
> 한국 ↔ 싱가포르 항로 전담 | 2026년 3월

## 1. 이 프로그램이 필요한 이유

콘솔사 오퍼레이터는 여러 화주의 소량 화물(LCL)을 모아서 하나의 컨테이너(FCL)로 합쳐 배에 싣는 일을 해요. 이 과정에서 견적, 부킹, 컨테이너 적재, 서류 처리, 마감 관리, 현장 소통을 모두 처리해야 하는데, 현재는 엑셀 + 카카오톡 + 이메일로 따로따로 처리하고 있어요.

CONSOLINK는 이 모든 게 한 곳에서 움직이는 프로그램입니다.

### 현재 vs CONSOLINK

| 현재 방식 | 문제점 | CONSOLINK로 해결 |
|-----------|--------|------------------|
| 엑셀로 화물/컨테이너 관리 | 같은 데이터를 여러 파일에 중복 입력 | 한 번 입력하면 모든 곳에 자동 반영 |
| 카톡으로 창고에 지시 | 지시 내용 빠뜨리거나 헷갈림 | 작업 지시서 자동 생성 -> 복사만 하면 끝 |
| 마감 시간을 머릿속으로 기억 | 마감 놓치면 대형 사고 | 카운트다운 + 색상 알림으로 절대 안 놓침 |
| 컨테이너 쌀 때 마진 수계산 | 수익성 판단이 늦음 | 화물 추가할 때마다 마진을 실시간 표시 |
| 적재 요구사항 머릿속 정리 | 충돌 나면 사고, 고민 시간 김 | 충돌 자동 감지 + 최적 배치 자동 추천 |

### 1.2 현장 전문가의 목소리 - 왜 기존 방식이 바뀌지 않았나?

실제 콘솔사 현장에서 10년 이상 근무한 오퍼레이터들의 피드백을 반영했습니다.

#### 현장 피드백 #1: CLP(적재 프로그램)가 있는데 왜 안 쓰나?

> **현직 콘솔 오퍼레이터:** "CLP 쓰면 많이 못 넣어서 실적이 안 나오거든. 프로그램은 안전율을 따지니까 물건을 70%밖에 못 채워. 근데 우리는 현실적으로 90% 이상 채워야 수익이 나."

**CONSOLINK의 해답:** 기존 CLP는 완벽한 자동화를 추구합니다. CONSOLINK는 다릅니다. 최종 판단은 사람이 하되, 시스템이 '가이드'를 제공합니다. 90% 채워도 되지만, 과적 위험 구간에 들어가면 경고를 띄워주는 거예요.

#### 현장 피드백 #2: 데이터와 현실이 항상 다르다

> **현직 콘솔 오퍼레이터:** "이게 우리가 보는 데이터랑 실제 현장 화물이 늘 차이가 나서 수정은 불가피함. 그래서 경험이 중요한 거야."

**CONSOLINK의 해답:** 맞아요, 데이터와 현장은 항상 다릅니다. 그래서 CONSOLINK는 '고정된 자동화'가 아니라 '유연한 가이드'입니다. 실시간으로 수정할 수 있고, 수정 이력이 모두 기록됩니다. 오히려 '어떤 화주의 화물이 얼마나 자주 실제와 다른지' 데이터가 쌓이면서 더 정확한 판단을 도울 수 있어요.

#### 현장 피드백 #3: 항구마다, 화물마다 다 달라

> **현직 콘솔 오퍼레이터:** "나라마다 CFS 장비도 다르고 할 수 있는게 달라. 싱가폴은 지게차 안 쓰고 손으로 하는 곳도 있어."

**CONSOLINK의 해답:** 항구별, CFS별 조건을 프로필로 저장할 수 있어요. 싱가포르 CFS는 수작업 위주, 부산 CFS는 지게차 사용 가능 등 다른 배치를 추천합니다.

#### 현장 피드백 #4: 그래도 가이드는 좋다

> **현직 콘솔 오퍼레이터:** "프로그램이 가이드 주는 건 좋을 듯? 유의미하면 팔릴 듯. 만들어서 구독료 받아라."

**핵심 인사이트:** 현장 전문가조차 '가이드형 도구'의 가치를 인정했습니다. 완전 자동화가 아니라, 경험 많은 오퍼레이터의 판단을 도와주는 도구 - 이것이 CONSOLINK의 정확한 포지션입니다.

### 1.3 기존 프로그램이 실패한 이유 5가지

| # | 기존 프로그램의 문제 | 현장의 반응 | CONSOLINK의 접근 |
|---|---------------------|-----------|-----------------|
| 1 | 정확한 cm 치수 요구 | 데이터가 부정확해서 의미없음 | CBM 비율 기반 Zone 배치 (현실적) |
| 2 | 안전율 70%로 제한 | 실적 안 나와서 안 씀 | 90%+ 가이드 + 위험 경고 (현실 반영) |
| 3 | 화주 간 충돌 미고려 | 콘솔은 다수 화주 혼적이 핵심 | 화주별 요구사항 충돌 자동 감지 |
| 4 | 비싼 라이선스 비용 | 중소 콘솔사는 감당 불가 | 합리적 구독 모델 (소규모 맞춤) |
| 5 | 한국어 미지원 / UX 복잡 | 배우는 데 시간 오래 걸림 | 한국어 완벽 지원 + 직관적 UI |

## 4. 기능 상세 설명

### 4.1 컨테이너 적재 계획 + 3D + 충돌감지 + 자동배치

> 콘솔사의 모든 업무는 '적재를 어떻게 할 거냐'에서 시작됩니다.

- 화물을 드래그해서 컨테이너에 배치
- CBM과 중량을 실시간으로 계산
- 화주별 요구사항(층간 금지, 위치 지정) 충돌 자동 감지
- AI가 최적 배치를 추천 (수익성 우선 또는 안정성 우선 선택 가능)
- 3D로 시각화해서 실제 모습을 미리 볼 수 있음

**컨테이너 규격:**

| 규격 | CBM | 중량(ton) | 내부 크기 |
|------|-----|-----------|-----------|
| 20ft FCL | 33 CBM | 25~28톤 | 5.9 x 2.35 x 2.38m |
| 40ft FCL | 67 CBM | 26~30톤 | 12.03 x 2.35 x 2.38m |

**Zone 기반 배치 (CONSOLINK만의 차별점):**

| Zone | 위치 | 특징 | 배치 기준 |
|------|------|------|-----------|
| ZONE A | 컨테이너 안쪽 (End) | 무겁고 먼저 넣는 화물 | 중량 우선 |
| ZONE B | 컨테이너 중간 (Middle) | 일반 화물 | CBM 효율 우선 |
| ZONE C | 컨테이너 문쪽 (Door) | 나중에 넣고 먼저 빼는 화물 | 하역 순서 우선 |

### 4.2 마감시간(Closing Time) 관리

> 놓치면 대형사고 - Docu/Cargo/VGM 3가지 마감을 카운트다운으로 관리

| 마감 종류 | 할 일 | 놓치면 |
|-----------|-------|--------|
| Docu Close | 선사에 B/L 제출 | 서류 늦으면 배편 취소 |
| Cargo Close | 화물을 CFS에 반입 | 화물 못 탄다 = 다음 배로 |
| VGM Close | 선사에 무게 신고 | 항만법 위반 - 페널티 |

카운트다운 색상: 48h+ 초록, 24~48h 주황, 24h 이내 빨강, 초과 시 깜빡임

### 4.3 돌발상황 대응 시스템

- 화물 지연 등록 시 시스템이 자동으로 다른 컨테이너 시뮬레이션
- 잘못된 배치 감지 시 수정안 제안
- 배 스케줄 변경 감지 시 영향받는 화물 자동 체크
- 각 시나리오별 영향 분석 (비용 변화, 마진 변화)

### 4.4 중량 초과 사전 경고

- CBM과 중량을 실시간으로 모니터링
- 컨테이너 최대 중량의 90% 도달 시 주의 경고
- 95% 이상 시 위험 경고
- VGM 신고 전에 최종 체크

### 4.5 투명성/감사 추적 시스템 [NEW] 🔒

> 콘솔사 비즈니스는 구조적으로 정보 비대칭 + 현장과 데이터의 괴리 + 수기 작업 의존이 겹쳐 있어 부정이 가능한 환경입니다.

**감지 가능한 부정 유형:**

| # | 부정 유형 | 어떻게 가능한가 | CONSOLINK 감지 방법 |
|---|----------|----------------|-------------------|
| 1 | 운임 마진 조작 | 영업사원이 거래처와 짜고 Selling Rate를 부풀려 차액을 나눠 가짐 | Buying/Selling 이력 자동 기록 + 평균 마진율 대비 이상치 알림 |
| 2 | 적재량 조작 | 실제보다 적게 보고하고 남는 공간에 무단 화물 적재 | 계획 vs 실제 CBM 자동 비교 + 차이 기록 |
| 3 | 실적 조작 | 가짜 Booking 후 취소, 물량 분산으로 실적 부풀리기 | Booking-선적-정산 전체 추적 + 취소율 자동 집계 |
| 4 | VGM 중량 허위신고 | 실제보다 가볍게 신고하여 운임 절약 (과적 위험) | 중량 모니터링 + 법적 한도 자동 체크 |
| 5 | 서류 내용 변조 | B/L, 패킹리스트 수량/내용 조작 | Booking 데이터 기반 자동 생성 + 불일치 감지 |

**감사 추적 시스템의 4가지 핵심 기능:**
1. 완전한 변경 이력: 누가, 언제, 무엇을, 왜 변경했는지 모든 기록 보존
2. 이상 거래 알림: 평균 대비 비정상적인 마진율, 취소율, 변경 빈도 자동 감지
3. 마진율 모니터링: 노선별/화주별 평균 마진율 대비 이상치 하이라이트
4. 데이터 무결성: 한번 기록된 데이터는 삭제 불가, 수정 시 원본 + 수정본 모두 보존

### 4.6 컨테이너 마진(수익) 계산

> 수익 = (화주들한테 받은 운임 합계) - (선사에 낸 FCL 운임)

| 항목 | 금액 (USD) | 설명 |
|------|-----------|------|
| FCL 운임 | $3,500 | 선사에 낼 비용 (40ft) |
| 화주A 운임 | $1,000 | CBM 15 |
| 화주B 운임 | $1,200 | CBM 18 |
| 화주C 운임 | $1,300 | CBM 20 |
| **합계 Selling** | **$3,500** | 수주 누적 |
| **마진** | **$0** | 손익분기점 - 더 받아야 함 |

### 4.7 원클릭 문서 자동생성

- CFS 작업 지시서 자동 생성
- Pre-alert 이메일 자동 생성
- 견적서 PDF 자동 생성
- B/L 드래프트 자동 생성

### 4.8 운임 계산 & 견적서

- R/T 자동 판정: CBM vs 중량 중 큰 것으로 자동 선택
- R/T(Revenue Ton) = Max(CBM, 중량 / 1.0)
- 최소 운임 적용, 통화 변환 (USD ↔ KRW)

### 4.9~4.17 나머지 기능

- **화주 신뢰도:** 지연 횟수, 평균 지연 시간 자동 기록 → 신뢰도 점수
- **과거 데이터 분석:** 유사 사례 추천
- **거래처 관리:** 화주 정보 DB + 거래이력
- **화물 추적:** 접수→CFS입고→선적→운송중→도착→통관
- **부킹/B/L:** 선사별 부킹 진행률, MBL/HBL 관리
- **스케줄 변경 감지:** 영향받는 컨테이너 자동 체크
- **하루 마감 체크리스트:** 퇴근 전 누락 방지
- **엑셀 데이터 업로드:** 과거 데이터 import

## 6. 한계 및 향후 확장

### 6.1 초기 버전의 한계

- 마감 알림은 브라우저가 열려 있을 때만 작동 (향후 모바일 앱 확장)
- 3D 시각화는 CBM 비율 기반 (정밀 cm 시뮬레이션 아님 - 의도적, 현장 피드백 반영)
- 환율은 수동 입력 (실시간 환율 연동 추후 확장)
- 화주 신뢰도는 데이터 30건 이상 쌓여야 정확
- 선박 스케줄 변경은 직접 입력 (선사 시스템 자동 연동 추후 확장)

### 6.2 향후 추가 가능 기능

- 모바일 앱, 카카오톡 알림봇
- 이메일 직접 발송, 정산서 관리
- 실시간 환율 연동, 선사 시스템 자동 연동
- 화물 사진 첨부 (클레임 대비)
- 다른 항로 추가 (미국, 유럽, 동남아)
- 고급 BI 분석, CFS별 프로필 저장

### 6.3 기술 스택

- Frontend: React, TypeScript, Tailwind CSS
- 3D: Three.js
- Backend: Node.js + Express
- Database: PostgreSQL + Prisma ORM
- AI/ML: Python + scikit-learn
- 배포: AWS or Azure
- 인증: JWT + bcrypt
- 상태관리: Zustand
- 감사 로그: PostgreSQL Audit Trail + 이벤트 소싱
