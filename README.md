> > ## 작업 규칙 (필수 준수)
> >
> > ### 1. 리더 체계
> > - **모든 작업은 Opus 리더가 총괄**한다
> > - - 리더가 작업을 분배하고, 각 팀원의 결과물을 검토한다
> >   - - 최종 merge/commit 결정은 리더만 한다
> >    
> >     - ### 2. 작업 시작 전
> >     - - 매 작업 시작 시 **팀원 전원이 참여하는 회의**를 진행한다
> >       - - 회의에서 요구사항 분석, 구현 방식, 역할 분담을 확정한다
> >         - - **충분한 논의 후에만 코딩을 시작**한다 (성급하게 시작하지 말 것)
> >          
> >           - ### 3. 교차 검증 (Cross-Review)
> >           - - 모든 코드는 **최소 2명 이상의 팀원이 교차 검증**한다
> >             - - 프론트엔드 코드 → Backend Lead + QA Reviewer 검증
> >               - - 백엔드 코드 → Frontend Lead + Security 검증
> >                 - - DB 변경 → DB Architect + Backend Lead 검증
> >                   - - **검증 없이 commit하지 않는다**
> >                    
> >                     - ### 4. 오류 제로 정책
> >                     - - 추측으로 코드 수정하지 않는다. **반드시 로그/수치 확인 후 수정**
> >                       - - 같은 문제 **3회 반복 시 즉시 멈추고** 근본 원인 분석 보고
> >                         - - 테스트 통과 없이 완료 보고하지 않는다
> >                           - - **실제 브라우저 기준 검증** (headless 테스트만으로 완료 판단 금지)
> >                            
> >                             - ### 5. 커밋 규칙
> >                             - - 커밋 메시지는 한글로 명확하게 작성
> >                               - - 기능 단위로 커밋 (하나의 커밋에 여러 기능 섞지 않기)
> >                                 - - 커밋 전 반드시 lint + 테스트 통과 확인
> >                                  
> >                                   - ### 6. 성능 관리
> >                                   - - 성능 문제는 **테스트 → 측정 → 수정 → 재측정** 루프로 진행
> >                                     - - 병목은 반드시 수치로 확인 후 보고
> >                                       - - 추측 기반 최적화 금지
> >                                        
> >                                         - ### 7. 매 작업 완료 후
> >                                         - - 서버 재시작 + curl + 브라우저 자동화 **교차 검증** 수행
> >                                           - - 테스트 없이 보고하지 않는다
> >                                             - - 결과물은 반드시 **실제 동작 확인 후** 커밋
> >
